## Top findings (priority order)

### C1. [疑似, high confidence] LS Newton Term 2/3 column mapping mismatches the residual's true dependencies on mixed-side *plain* elements
- **Where**: `pyfp3d/kernels/cut_assembly.py:437-457` (Term 2/3 scatter, cols =
  `dof[e]`/`dof[u3]`); root cause `pyfp3d/wake/multivalued.py:494-520`
  (`_side_dofvecs` gives plain elements main connectivity) +
  `cut_assembly.py:154-155` (comment "A plain element is single-sided" — false
  for beyond-tip elements).
- **What**: on 3D meshes there are uncut mixed-side elements (beyond-tip
  elements rejected by the spanwise clip, and elements the wake-surface
  extension passes through upstream of the TE). `mass_conservation_coo`
  assembles them with the **side-field density** onto main DOFs. The residual
  therefore depends on aux DOFs of cut nodes through `side_potentials`, but
  Term 2/3 scatter all sensitivities to **main columns**, with gradient
  factors from the side field while the residual multiplies the main-field
  ∇φ. For these elements J ≠ dR/dφ: (a) real aux-column derivative terms are
  missing, (b) spurious terms are added to main columns of "−" nodes,
  (c) "+" node columns get mismatched gradient factors.
- **Why the gates can't see it**: quasi-2D meshes have no beyond-tip elements
  (so B6's FD gate `test_b6_newton` structurally misses the class), and the
  B15 M6 gates are convergence gates, not FD gates. `s_e = ρ'_e ≠ 0` holds
  subsonically too, so the terms really are emitted. Impact unmeasured — the
  Newton degrades to quasi-Newton and can still converge (GB15.4's M6 numbers
  do not rule this out). The affected set is provably non-empty on M6 medium:
  `multivalued.py:684-703` documents "side 1.532 vs main 0.309".
- **Suggested verification / fix**: JVP FD check of `LSNewtonSystem` on a 3D
  mesh with beyond-tip elements, probe directions on DOFs adjacent to the
  beyond-tip strip. Fix either by per-node side-aware column mapping
  (+ → main, − cut → aux), or by using the main-field density for these
  elements (consistent with `element_mach2`'s "main" reading).

### C2. [疑似] conforming `newton.py` fail-fast compares across "selection epochs" (fixed on LS, not backported)
- **Where**: `pyfp3d/solve/newton.py:638, 690, 723`; contrast fixed
  `pyfp3d/solve/newton_ls.py:549, 654, 703`.
- **What**: `r_level_best` only min-updates and is never reset on
  freeze/refresh/revert. After a frozen phase converges to ~1e-11, an honesty
  live re-evaluation legitimately jumps back to the live residual scale; if
  that floor is > 1e-3, `r_norm > 100*r_level_best and r_norm > 1e-3` fires
  immediately and kills a healthy freeze-refresh cycle. This is exactly
  design_track_b §14.5 pitfall #3, measured on the LS path (M6 medium M0.65)
  and fixed there with `r_best = np.inf` at epoch changes. The conforming
  path has not bitten only because 2.5D floors (~1e-8…1e-5) sit below the
  1e-3 absolute floor — G9.1's conforming fine-mesh region is precisely the
  high-floor regime.

### C3. [疑似] conforming `newton.py` lacks `freeze_max_reverts` permanent disarm (LS's B15 safety net not backported)
- **Where**: `pyfp3d/solve/newton.py:698-716` (revert only sets
  `freeze_cooldown = 5`); contrast `pyfp3d/solve/newton_ls.py:555-556`.
- **What**: the conforming driver has no cap on the
  freeze→diverge→revert→cooldown→re-arm loop; a repeatedly diverging freeze
  burns ≥6 steps per round and can exhaust `n_newton_max` (no wrong result,
  but a wasted level budget and possibly a ramp killed that the live path
  would have saved). LS added `freeze_max_reverts` (default 3, permanent
  disarm: "freezing can only help, never cost convergence") in B15.
  `n_freeze_reverts` is counted/reported in newton.py but never used.

### C4. [确认 defect — for external meshes] mesh reader silently drops triangles of unnamed physical groups
- **Where**: `pyfp3d/mesh/reader.py:146-152`.
- **What**: when a .msh has `$PhysicalNames` naming only *some* surface
  groups, triangles of unnamed groups never enter `boundary_faces` (the
  `"all_triangles"` fallback fires only when there is no physical data at
  all). **Runtime-reproduced by the reviewer** (minimal mesh: unnamed tag 7 +
  named "wall" tag 1 → tag-7 triangles vanish). Consequence chain for an
  external gmsh mesh with wall/wake named but symmetry/fuselage unnamed:
  symmetry-plane triangles lost → `wake_cut._sheet_free_edge_nodes` misjudges
  the sheet's symmetry edge as an interior free edge → root-station nodes not
  duplicated → **Γ(root) silently pinned to 0** (wrong aerodynamics, no
  error). All in-repo meshes name everything, so this is latent — but the
  reader is the entry point for external meshes.

### C5. [确认 loud failure on legal input] reader crashes on unnamed physical *volume* tags
- **Where**: `pyfp3d/mesh/reader.py:167-174` + `Mesh.validate()`.
- **What**: a legal .msh without `$PhysicalNames` but with physical volume
  tags gets `element_tags` = raw ids (e.g. [5,5]) while `tag_names` stays
  `["bulk"]` (rebuild skipped by `if volume_entries:`), so
  `Mesh.validate()`'s `element_tags < len(tag_names)` raises AssertionError.
  Loud, not silent, but a crash on legal input; placeholder names should be
  rebuilt as on the surface side. Runtime-reproduced by the reviewer.

### C6. [疑问] far-field wake master selection uses exact float equality `dy == 0.0`
- **Where**: `pyfp3d/constraints/dirichlet.py:81` — `on_cut = (dy == 0.0) &
  (dx > 0.0)`.
- **What**: the θ=2π branch for far-field wake master nodes relies on the y
  coordinate being bit-exact 0; the generator only asserts |y| < 1e-7·r_far
  (sphere∩sheet intersection nodes come from OCC numerical intersection). A
  master node at y=+1e-9 escapes the mask, gets the upper-branch value → its
  Dirichlet data is off by one Γ (sawtooth of order Γ along the wake line),
  and the eliminated slave = master+Γ inherits it. P5's branch-ray fix
  evidence shows committed meshes have y exactly 0, but this is a fragile
  dependence on gmsh behavior. More robust: mask on `is_master_wake & (dx>0)`
  only — cut membership already implies identity.

### C7. [疑似 latent] no guard when a TE node has no aux DOF
- **Where**: `pyfp3d/wake/multivalued.py:315-317` +
  `pyfp3d/kernels/cut_assembly.py:332`.
- **What**: if a TE node is not a node of any cut element
  (`ext_dof_of_node = −1`): `reroute[te_aux] = cm.te_nodes` writes
  `reroute[−1]` (i.e. `reroute[n_total−1]`), silently redirecting the last
  aux dof's mass-conservation row onto the TE main row and leaving that aux
  dof without an equation; `te_kutta_coo` also emits row=−1 triplets (newer
  scipy raises, far from the root cause). B1's gate locks "TE nodes ⊆ cut
  nodes" on committed meshes, but `CutElementMap.__init__` itself has no such
  assert. Suggest `assert (ext_dof_of_node[te_nodes] >= 0).all()`.
- Related corner (from the kernels review): TE nodes are only shifted to the
  "+" side within `|s_raw| < 1e-6·h` while TE detection uses `1e-3·h`
  (`wake/cut_elements.py:128-143`); a TE node landing in (−1e-3, −1e-6)·h
  would be classed "−" and `ext_dof_of_node[t]` (the *upper* value) would be
  used as the *lower* TE reference, inverting López fig. 3.6c / D11 intent.
  No shipped mesh triggers it; suggest forcing TE nodes into shift_mask or
  asserting `node_side[te_nodes] == 1`.

### T1. [gate weakened in test] B3's test bound is 7× looser than the gate criterion
- **Where**: `tests/test_b3_lifting.py:124` asserts `abs(g3-g0)/g0 < 0.02`,
  while `docs/design_track_b.md`'s B3 gate says Γ within **0.3%** of the

### T2. [test bug, minor] dropped `validate_coloring` return value — assertion is a no-op
- **Where**: `tests/test_mesh_coloring.py:59` calls
  `validate_coloring(elements, colors)` without `assert` (contrast :43);
  `pyfp3d/mesh/coloring.py:148` returns bool. Coverage loss is small (cube

### P1. [API contract] `unified.section_cp` silently drops `gamma` on the conforming branch
- **Where**: `pyfp3d/post/unified.py:120-123` +
  `pyfp3d/post/section_cut.py:266,273-297` — the conforming chain
  (`section_cp_curve`/`wall_cp_curve`/`_wall_section_points`) has no gamma
  parameter and uses GAMMA=1.4 internally, while the LS branch
  (`section_cp_curve_levelset`) passes gamma through. A caller passing γ≠1.4
  gets different physics per branch with no warning. Dormant today (all cases
  γ=1.4) but it breaks the very comparability contract `unified` exists for.

### P2. [robustness, low] post-processing inherits the solver's vacuum floor — Cp saturates silently instead of erroring
- **Where**: `pyfp3d/physics/isentropic.py:56-58` via
  `pyfp3d/post/surface.py:265-286`.
- **What**: density's vacuum guard (base clamped to 1e-10, meant to protect
  the Picard matrix) is inherited by the post path: a super-vacuum wall
  gradient (q² > 1+2/((γ−1)M∞²), ≈8.1 at M0.84) yields Cp silently
  saturated at the vacuum value ≈ −2/(γM∞²) instead of NaN/error. The
  solver's q² limiter (M_cap=3) constrains element q² but not the wall
  triangles' own in-plane P1 gradients. Measured TE spike residual ~0.05 —
  nowhere near triggering. Also: `validate_physics_bounds`
  (`isentropic.py:370-371`) errors at q > 2.0, contradicting legal
  limiter-capped fields up to q ≈ 2.28 (only caller is test_v0_freestream at
  M0.5, but it will false-positive if wired into transonic monitoring).
